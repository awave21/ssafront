## SQNS интеграция — фронтендовая спецификация

### 1. Краткий обзор
- На странице карточки агента добавляем кнопку **«Включить SQNS»**. По нажатию открывается модальное окно c формой `host` + `apiKey` (вводится один раз, не сохраняется в клиентском state) и, опционально, `token_path`, если CRM требует отличного эндпоинта.
- После успешного запроса форма закрывается, с карточки исчезает ошибочный статус и появляется блок со статусом интеграции (`sqns_enabled`, `sqns_status`, `last_sync_at`, `sqns_error`).
- Управление визитами остаётся на стороне агента/интеграции — фронт только показывает, что SQNS подключён и где получить данные для тулзов.

### 2. Данные для формы и статуса

#### Вариант A: включение по email и паролю (рекомендуется)

Форма отправляет `POST /api/v1/agents/{agent_id}/sqns/enable-by-password` (см. [схему](sqns-enable-by-password-schema.md)):

```json
{
  "host": "crmexchange.1denta.ru",
  "email": "user@example.com",
  "password": "user_password",
  "defaultResourceId": 123
}
```

- `host` — хост SQNS без схемы (бэкенд подставит `https://`) или с `https://`.  
- `email`, `password` — логин и пароль; бэкенд сам запрашивает токен (POST `{host}/api/v2/auth`) и сохраняет его.  
- `defaultResourceId` — опционально.

Бэкенд: запрос `POST {host}/api/v2/auth` с телом `{email, password, defaultResourceId?}`. Ответ: массив `[{ "status": "success", "token": "...", "user": {...} }]`; из первого элемента берётся `token`, сохраняется в credential с `direct_bearer: true` и привязывается к агенту. Пароль на фронте не сохраняется.

#### Вариант B: включение по API-ключу (legacy)

`POST /agents/{agent_id}/sqns/enable`:

```json
{
  "host": "https://crmexchange.1denta.ru",
  "api_key": "секретный ключ",
  "token_path": "/api/v2/auth/token",
  "direct_bearer": false
}
```

- `host` — `https` URL.  
- `api_key` — обязательный, после отправки очищаем поле.  
- `token_path` — необязательно.  
- `direct_bearer` — если `true`, `api_key` считается готовым Bearer-токеном и не обменивается по `token_path`.

После включения показываем карточку статуса, читаемые поля от backend:

```json
{
  "sqns_enabled": true,
  "sqns_configured": true,
  "sqns_host": "https://crmexchange.1denta.ru",
  "sqns_status": "ok", // ok | error | disabled
  "sqns_last_sync_at": "2026-01-25T12:00:00Z",
  "sqns_error": null
}
```

- `sqns_enabled` → просто переключатель в UI.  
- `sqns_status` / `sqns_error` → показываем цветовой тег (зелёный | красный | нейтральный) и текст ошибки (`"Токен просрочен"`).  
- `sqns_last_sync_at` → время последнего обращения к CRM (обновляется при каждом успешном запросе по тулзам).  
- `sqns_configured` → можно использовать, чтобы не показывать форму повторно до отключения.

Добавьте кнопку **«Отключить SQNS»**, которая вызывает `POST /agents/{agent_id}/sqns/disable`. После отключения показывайте статус `disabled` и текст «Интеграция отключена».

### 3. Контракты по данным (для подсказок агенту)
Фронт предоставляет информацию об источниках данных, которые нужны тулзам агентов. Каждая подсказка может ссылаться на эндпоинт и нужные параметры:

| Параметр | Описание | Источник API |
| --- | --- | --- |
| `serviceIds` | Идентификаторы услуг, которые агент предлагает | `GET /agents/{agent_id}/sqns/services` |
| `resourceId` | Идентификатор специалиста/ресурса | `GET /agents/{agent_id}/sqns/resources` |
| `datetime` | Выбор слота — брать из `GET /agents/{agent_id}/sqns/slots?date=...&resourceId=...` |
| `clientPhone` | lookup по телефону клиента (формат `7XXXXXXXXXX`) | `GET /agents/{agent_id}/sqns/client/{phone}` |
| `comment` | Текстовый комментарий | формируется из диалога, не зависит от CRM |


#### Ответы серверных эндпоинтов

- `GET /agents/{agent_id}/sqns/resources` → `{ "resources": [ { "id": 15, "name": "Стоматолог", ... } ] }`  
- `GET /agents/{agent_id}/sqns/services` → `{ "services": [ { "id": 123, "name": "...", "duration": 60 } ] }`  
- `GET /agents/{agent_id}/sqns/slots?resourceId=15&date=2026-01-25&serviceIds=123` → `{ "slots": { "resource": {...}, "slots": [ { "datetime": "...", "isAvailable": true } ] } }`  
- `GET /agents/{agent_id}/sqns/visits?dateFrom=2026-01-25&dateTill=2026-01-25` → `{ "visits": [...] }`  
- `GET /agents/{agent_id}/sqns/client/79991234567` → `{ "client": { "name": "...", "phone": "7999..." } }`

Фронт может показывать эти объекты в виде списка/карточки, чтобы агент видел, откуда берутся `serviceIds`/`resourceId`/`datetime`. В поле `slots` можно отметить, что данные приходят из CRM и использовать их для валидации на стороне агента.

### 4. UX и ошибки

#### Ошибки, которые нужно показывать пользователю
- **Не удалось получить токен:** `POST /agents/{agent_id}/sqns/enable` возвращает 502, `sqns_error` содержит текст. Покажите модалку/тултип «Хост/ключ недоступны».
- **Нет слотов:** `GET /agents/{agent_id}/sqns/slots` возвращает `{ "slots": { "slots": [] } }` — показывайте сообщение «Свободных слотов нет».
- **Конфликт визита:** `GET /agents/{agent_id}/sqns/visits` или тулз агента показывает, что визит "клиент пришел" — обозначьте, что нужно выбрать другую дату.
- **Host недоступен / неверный ключ:** любая ошибка `SQNSClientError` отображается через `sqns_status=error` и `sqns_error`.

#### Безопасность и UX
- Все запросы идут через бекенд, поэтому фронт не управляет `Bearer`-токеном — он всегда приходит автоматом из `Authorization: Bearer` в заголовках (об этом сообщается Архитектору UI).  
- При вводе `apiKey` в модалке показывайте маску (●●●) и кнопку "Показать", но после отправки очищайте поле.  
- Показывайте подсказку: «API-контакт хранится зашифрованно и никогда не выводится повторно».

### 5. Следующие шаги на фронте

1. Поддерживать блок статуса интеграции на карточке агента (кнопка включения/отключения, индикатор `sqns_enabled`/`sqns_status`).  
2. Рисовать список источников данных (`serviceIds`, `resourceId`, `datetime`, `clientPhone`) с ссылками на эндпоинты, чтобы агент понимал, откуда брать значения.  
3. Показывать ошибки, возвращаемые от SQNS, в `sqns_error`.  
4. Обновлять данные `resources/services/slots` перед запуском тулза, чтобы агент использовал актуальную информацию из CRM.  
5. Не создавать отдельных форм управления визитами: интерфейс только показывает, что SQNS активна и какие данные нужны для тулзов агента.
